## Como reportar bug

<Steps>
  <Step title="Confere se já não existe">
    Procura em **Issues** do repo correspondente antes de abrir. Bug duplicado a gente fecha apontando pro original.
  </Step>
  <Step title="Reproduz isolado">
    Snippet mínimo que reproduz o bug — não cola código de produção inteiro. "Stack overflow style" funciona.
  </Step>
  <Step title="Abre issue com template">
    Cada repo tem template de bug report. Preencha:
    - SDK version (`@ararahq/sdk@1.8.0`)
    - Runtime version (Node 20, PHP 8.2, etc)
    - Sistema operacional
    - Comportamento esperado vs observado
    - Snippet reproduzível
  </Step>
  <Step title="Aguarda triage">
    Triage em **2 dias úteis**. Bugs críticos (segurança, perda de dados) priorizados.
  </Step>
</Steps>

## Como sugerir feature

Discussão antes de PR. Abra **Discussion** (não Issue) descrevendo:

- Problema que você está tentando resolver
- Solução que você imagina
- Alternativas consideradas
- Quem se beneficia

Features pequenas (helper method, validação) viram PR direto. Features grandes (novo recurso, mudança de API) precisam alinhamento prévio.

## Pull Requests

### Antes de codar

- Abra issue/discussion primeiro pra alinhar abordagem
- Comente "vou pegar essa" pra evitar trabalho duplicado
- Fork → branch (`feat/xxx` ou `fix/xxx`) → PR

### O que esperamos

| Critério | Requisito |
|---|---|
| **Testes** | Funcionalidade nova vem com teste. Bug fix vem com teste que falha antes do fix. |
| **Lint** | `npm run lint`, `ruff`, `composer lint`, `mvn checkstyle` — zero warning |
| **Commit message** | `tipo(escopo): descrição` (ex: `feat(messages): add interactive payload type`) |
| **CHANGELOG** | Adicione entrada em `CHANGELOG.md` sob `## [Unreleased]` |
| **Docs** | Se mudou API pública, atualize README do SDK |

### Review

- 1 maintainer aprova → merge
- Tempo médio de review: **48-72h**
- Sem feedback em 1 semana? Marca o maintainer no PR

### Tamanho do PR

Pequenos e focados. PR grande (> 500 linhas) divide em série. Difícil revisar mega-PRs.

## Documentação

Esta página, todas as outras desta doc, e as receitas práticas vivem em [github.com/ararahq/docs](https://github.com/ararahq/docs). É Mintlify-based.

Pra contribuir com a doc:

```bash
git clone https://github.com/ararahq/docs
cd docs
npx mintlify dev  # roda local em http://localhost:3000
```

Edite o `.mdx` correspondente, dê preview local, abre PR.

### Padrões

- **Tom**: dev-first BR, sem hype, honesto sobre limitações
- **Slugs**: inglês curto (`/concepts/whatsapp`, não `/conceitos/`)
- **Snippets**: validados contra código real, não inventados
- **Frontmatter**: title + description + keywords obrigatórios
- **Idioma**: PT-BR salvo termos técnicos consolidados (webhook, payload, retry)

## Code of conduct

Comportamento esperado:

- **Respeito**: assuma boa-fé. Crítica ao código, não à pessoa.
- **Inclusão**: sem comentários racistas, sexistas, homofóbicos, etc. Zero tolerância.
- **Construtividade**: "isso não funciona" não ajuda. "isso não funciona porque X, sugiro Y" sim.

Violação → ban do org GitHub. Reportar a [conduct@ararahq.com](mailto:conduct@ararahq.com).

## Reconhecimento

Top contributors aparecem no README de cada repo. Contribuições significativas viram caso na newsletter (com permissão).

Não pagamos por contribuição (ainda) mas qualquer PR mergeado entra pro registro público.
