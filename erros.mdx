A maioria dos erros da Arara segue um envelope JSON com `code` legível por máquina. Rotas mais antigas ainda retornam um envelope simplificado sem `code` — abaixo cada formato e quando esperar cada um.

## Formato canônico

```json
{
  "error": {
    "code": "PLAN_FEATURE_LOCKED",
    "message": "Essa feature está liberada a partir do plano BUSINESS.",
    "feature": "campaigns",
    "currentPlan": "STARTER",
    "upgradeTo": "BUSINESS"
  }
}
```

| Campo | Tipo | Descrição |
|---|---|---|
| `error.code` | string | Código estável (constante). Use em `if`/`switch`. |
| `error.message` | string | Descrição em português, pra exibir ao usuário. |
| Campos extras | varia | Cada `code` pode trazer contexto específico (planos, limites, currentStatus, etc). |

## Formato simplificado (sem `code`)

Algumas rotas — autenticação (filter do Spring), validação de body (`@Valid`), e exceptions genéricas — retornam:

```json
{
  "error": "Unauthorized",
  "message": "Invalid API Key format"
}
```

Use o `status` HTTP pra ramificar, não `error.code` (pode ser `null` aqui). Padronização está em andamento; eventualmente todas as rotas vão emitir `error.code`.

## Códigos com envelope canônico

### Plano e conta

| Código | HTTP | Quando |
|---|---|---|
| `PLAN_FEATURE_LOCKED` | 403 | Feature exige plano superior — payload traz `feature`, `currentPlan`, `upgradeTo`. |
| `PLAN_LIMIT_REACHED` | 403 | Limite do plano atingido — payload traz `feature`, `current`, `max`. |
| `ACCOUNT_NOT_ACTIVATED` | 403 | Onboarding incompleto — payload traz `currentStatus`. Bloqueia criação de chave LIVE. |
| `NO_DEDICATED_NUMBER` | 403 | Operação exige número WhatsApp dedicado. |
| `ORGANIZATION_SUSPENDED` | 429 | Org suspensa por taxa de falhas alta. |
| `NUMBER_BLOCKED` | 422 | Destinatário em opt-out ou número bloqueado preventivamente. |

### Recovery

| Código | HTTP | Quando |
|---|---|---|
| `RECOVERY_ENDPOINT_NOT_FOUND` | 404 | Endpoint de recovery não encontrado. |
| `RECOVERY_ENDPOINT_INACTIVE` | 403 | Endpoint existe mas está `active: false`. |
| `RECOVERY_EVENT_INVALID` | 400 | `eventType` ou `period` fora dos aceitos. |
| `RECOVERY_HMAC_INVALID` | 401 | Assinatura HMAC do parceiro não confere. |
| `RECOVERY_RATE_LIMITED` | 429 | Excesso de eventos por minuto — header `Retry-After: 60`. |
| `RECOVERY_CONFIG_NOT_FOUND` | 404 | Sem config pra esse `eventType`. |
| `RECOVERY_TEMPLATE_MISSING` | 412 | Config existe mas o template vinculado foi removido. |

### OAuth device flow

| Código | HTTP | Quando |
|---|---|---|
| `invalid_grant` | 400 | Device code inválido. |
| `authorization_pending` | 400 | Usuário ainda não aprovou no browser. |
| `slow_down` | 400 | Cliente CLI fazendo polling muito rápido. |
| `access_denied` | 400 | Usuário negou. |
| `expired_token` | 400 | Device code expirou — refaça `/oauth/device/code`. |
| `device_code_not_found` | 404 | Code não existe. |
| `device_code_already_used` | 409 | Code já foi trocado por access token. |

### Genéricos

| Código | HTTP | Quando |
|---|---|---|
| `NOT_FOUND` | 404 | Endpoint inexistente (rota sem handler). |
| `NOTIFICATION_NOT_FOUND` | 404 | Notification in-app não existe. |

## Erros sem `code` estruturado

Esses ainda saem com `{ error: "...", message: "..." }` (sem `code`). Use o status HTTP:

| HTTP | Origem típica |
|---|---|
| `400` | Validação de body (`@Valid` falhou). `message` traz a primeira violação. |
| `401` | API key ausente, malformada, revogada ou IP fora da allowlist. |
| `402` | Saldo da wallet insuficiente. Vem com `diagnostic` traduzindo o erro. |
| `403` | Acesso negado (escopo de chave, IP bloqueado, security exception). |
| `404` | Recurso não existe (link, mensagem, contato, etc — alguns ainda sem `code`). |
| `409` | Conflito de estado (`IllegalStateException`). |
| `422` | Regra de negócio violada (`BusinessException` — fora da janela 24h, template não existe, variáveis incompatíveis). Vem com `diagnostic`. |
| `500` | Erro interno não tratado. |

## Diagnóstico WhatsApp (`failureReason`)

Mensagens com `status: failed` trazem `failureReason` traduzido da Meta:

| `failureReason` | Diagnóstico | Ação |
|---|---|---|
| `63016` | Janela de 24h fechada | Use template HSM pra reabrir |
| `insufficient_funds` | Saldo esgotado | Recarga ou auto-recharge |
| `21211` | Número inválido ou sem WhatsApp | Confere DDI + DDD + dígito |
| `63024` | Template não encontrado / não aprovado | Verifica status no dashboard |
| `invalid_template_params` | Variáveis incompatíveis | Confere número/ordem das vars |
| `generic` | Erro genérico Meta | Retry em alguns minutos |

Consulta detalhe completo:

```bash
curl https://api.ararahq.com/v1/messages/ara_msg_xxx \
  -H "Authorization: Bearer ara_live_xxx"
```

## Retry automático

A Arara retenta automaticamente:

- **5xx upstream Meta**: backoff exponencial, até 5 tentativas em ~1h
- **429 rate limit Meta**: respeita `Retry-After`
- **HMAC inválido em recovery**: 401, sem retry — o parceiro precisa corrigir a assinatura

Mensagem só fica em `status: failed` depois que todos os retries falharam. Você **não precisa** retentar do seu lado pra esses casos.

Pra **4xx que não 429**, a Arara não retenta — corrigir antes (chave inválida, template ausente, número inválido).

## Estratégia de tratamento no cliente

```typescript
try {
  await sdk.messages.send({ ... });
} catch (err) {
  if (err.statusCode === 429) {
    await sleep((err.retryAfter ?? 60) * 1000);
    retry();
  } else if (err.error?.code === 'PLAN_FEATURE_LOCKED' || err.error?.code === 'PLAN_LIMIT_REACHED') {
    notifyAdmin(`Upgrade necessário: ${err.error.upgradeTo}`);
  } else if (err.error?.code === 'ACCOUNT_NOT_ACTIVATED') {
    openOnboardingWizard();
  } else if (err.statusCode === 402) {
    notifyAdmin('Saldo insuficiente — recarregar wallet');
  } else if (err.statusCode === 422) {
    queueForRetry();
  } else if (err.statusCode >= 500) {
    notifyOps(err);
  } else {
    throw err;
  }
}
```
