A API da Arara é REST padrão, JSON in/out, autenticação Bearer. Esta página cobre **convenções gerais** — formato, paginação, idempotência, headers. Endpoints específicos têm sua página.

## Base URL

```
https://api.ararahq.com
```

Não há SemVer no path. Mudanças incompatíveis viram **endpoint novo** (ex: `/v1/messages/batch` adicionado sem quebrar `/v1/messages`). Deprecations são anunciadas no [changelog](/changelog) com ≥ 6 meses de aviso antes de remover.

## Autenticação

A maioria dos endpoints aceita **Bearer API key**:

```
Authorization: Bearer ara_live_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

Endpoints de **sessão de usuário** (`/auth/*`, `/users/*`, `/v1/profile/*`) usam **Firebase JWT** em vez de API key. Veja [Autenticação](/api-reference/authentication) pra detalhes.

## Formato de resposta

Sempre **JSON**. Datas em **ISO 8601 (UTC)**.

### Objeto único

```json
{
  "id": "ara_msg_k9Bf2XqP",
  "status": "PENDING",
  "mode": "LIVE",
  "sender": "whatsapp:+5511888888888",
  "receiver": "whatsapp:+5511999999999"
}
```

### Lista paginada

Toda lista segue o mesmo envelope:

```json
{
  "data": [
    { "id": "ara_tpl_abc", "name": "pedido_confirmado" }
  ],
  "pagination": {
    "page": 0,
    "size": 20,
    "totalElements": 150,
    "totalPages": 8
  }
}
```

### Erro

Formato canônico:

```json
{
  "error": {
    "code": "INSUFFICIENT_FUNDS",
    "message": "Your wallet does not have enough credits for this operation.",
    "details": {}
  }
}
```

Lista completa de códigos em [Erros](/erros).

## Paginação

Endpoints que retornam lista aceitam:

| Param | Tipo | Default | Descrição |
|---|---|---|---|
| `page` | integer | `0` | Página, zero-indexed. |
| `size` | integer | `20` | Itens por página, máximo `100`. |

```bash
curl "https://api.ararahq.com/v1/messages?page=2&size=50" \
  -H "Authorization: Bearer ara_live_xxx"
```

## Idempotência

POSTs que criam recursos críticos aceitam header `Idempotency-Key`:

```bash
curl -X POST https://api.ararahq.com/v1/messages \
  -H "Authorization: Bearer ara_live_xxx" \
  -H "Idempotency-Key: order-confirm-abc123" \
  -H "Content-Type: application/json" \
  -d '{ "receiver": "...", "templateName": "...", "templateVariables": [...] }'
```

Comportamento:

- Mesma chave + mesmo body em 24h → retorna **resposta original**, não cria duplicata
- Mesma chave + body diferente → conflito (409)
- Sem chave em endpoint que **exige** → 400

Endpoints:

| Endpoint | Header |
|---|---|
| `POST /v1/messages` | opcional (recomendado) |
| `POST /v1/messages/batch` | opcional (recomendado) |
| `POST /v1/campaigns` | **obrigatório** |
| `POST /v1/brain/ingest-url` | opcional |
| `POST /v1/brain/ingest-pdf` | opcional |

## Rate limits

Limite global: **100 requisições / segundo / API key**.

Quando exceder, resposta `429 Too Many Requests` com headers:

```
HTTP/1.1 429 Too Many Requests
Retry-After: 3
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1716040331
```

Respeite `Retry-After`. SDKs oficiais já fazem retry com backoff exponencial automaticamente.

## Status codes HTTP

| Código | Significado | Quando |
|---|---|---|
| `200` | OK | Sucesso com body (consultas, listings) |
| `201` | Created | Recurso criado (template, campanha, contato) |
| `202` | Accepted | Aceito pra processamento assíncrono (mensagem na fila) |
| `204` | No Content | Sucesso sem body (delete, revoke) |
| `400` | Bad Request | Input malformado |
| `401` | Unauthorized | API key ausente ou inválida |
| `402` | Payment Required | Saldo de créditos insuficiente |
| `403` | Forbidden | Sem permissão pro recurso |
| `404` | Not Found | Recurso não existe |
| `409` | Conflict | Conflito (duplicata, idempotency mismatch, estado inválido) |
| `422` | Unprocessable Entity | Input válido mas regra de negócio quebrada (template não aprovado, janela fechada) |
| `429` | Too Many Requests | Rate limit excedido |
| `500` | Internal Server Error | Erro interno — retry seguro |
| `502` | Bad Gateway | Provider upstream (Meta, AbacatePay) instável — retry seguro |
| `503` | Service Unavailable | Indisponível — retry seguro |

## Convenções

### Telefone

Formato **E.164** com prefixo `whatsapp:`:

```
whatsapp:+5511999999999
```

Em retorno de webhook inbound, sem prefixo:

```json
{ "from": "+5511999999999" }
```

### IDs

Prefixados por tipo, Base62, 7 chars depois do prefixo:

| Prefixo | Tipo |
|---|---|
| `ara_msg_` | Mensagem |
| `ara_tpl_` | Template |
| `ara_cmp_` | Campanha |
| `ara_key_` | API key |
| `ara_num_` | Número |
| `ara_org_` | Organização |
| `ara_user_` | Usuário |
| `ara_conv_` | Conversa |

### Valores monetários

Em **centavos**, sempre inteiros, sempre BRL salvo indicação contrária:

```json
{ "amountCents": 19990 }   // R$ 199,90
```

### Booleanos e enums

Booleanos: `true`/`false` minúsculo. Enums em UPPERCASE: `LIVE`, `MARKETING`, `UTILITY`.

## CORS

A API aceita CORS de qualquer origem em `OPTIONS` (preflight), mas **rejeita request real do navegador** (CORS `403`). API key não pode ir pro browser — sempre proxy via seu backend.
