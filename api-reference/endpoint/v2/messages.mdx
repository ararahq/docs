O endpoint mais usado. Aceita template HSM (qualquer hora) ou texto livre (dentro da janela 24h). Suporta lote de até 1000 mensagens, agendamento, mídia, mensagem interativa e localização.

**Base URL:** `https://api.ararahq.com`

---

## Enviar mensagem

`POST /v1/messages`

<CodeGroup>
```bash cURL
curl -X POST https://api.ararahq.com/v1/messages \
  -H "Authorization: Bearer ara_live_xxx" \
  -H "Content-Type: application/json" \
  -d '{
    "receiver": "whatsapp:+5511999999999",
    "templateName": "pedido_confirmado",
    "templateVariables": ["João", "ARA-12345"]
  }'
```

```typescript Node.js
await sdk.messages.send({
  receiver: 'whatsapp:+5511999999999',
  templateName: 'pedido_confirmado',
  templateVariables: ['João', 'ARA-12345'],
});
```

```python Python
client.messages.send(SendMessageRequest(
    receiver="whatsapp:+5511999999999",
    template_name="pedido_confirmado",
    template_variables=["João", "ARA-12345"],
))
```

```php PHP
$sdk->messages->send(
    receiver: 'whatsapp:+5511999999999',
    templateName: 'pedido_confirmado',
    variables: ['João', 'ARA-12345'],
);
```

```java Java
arara.getMessages().send(SendMessageRequest.builder()
    .receiver("whatsapp:+5511999999999")
    .templateName("pedido_confirmado")
    .templateVariables(List.of("João", "ARA-12345"))
    .build());
```
</CodeGroup>

Resposta (`202 Accepted`):

```json
{
  "id": "ara_msg_k9Bf2XqP",
  "status": "PENDING",
  "mode": "LIVE",
  "sender": "whatsapp:+5511888888888",
  "receiver": "whatsapp:+5511999999999",
  "cost": null
}
```

`cost` vem `null` no envio inicial e é populado depois que a Meta confirma a entrega e a fatura é calculada. Use o `id` pra consultar status atualizado ou casar com eventos de webhook.

## Texto livre (janela 24h)

Cliente respondeu nas últimas 24h? Você pode mandar texto livre em vez de template:

```bash
curl -X POST https://api.ararahq.com/v1/messages \
  -H "Authorization: Bearer ara_live_xxx" \
  -d '{
    "receiver": "whatsapp:+5511999999999",
    "body": "Já vou te responder, só preciso checar uma coisa."
  }'
```

Antes de mandar, **valida a janela**:

```bash
curl -X POST https://api.ararahq.com/v1/conversations/window-status \
  -H "Authorization: Bearer ara_live_xxx" \
  -d '{ "phones": ["+5511999999999"] }'
```

Se `status: CLOSED`, manda template em vez de body. Detalhes em [Janela 24h](/concepts/24h-window).

## Template com mídia (header)

URL da mídia vai como **primeiro elemento** de `templateVariables`:

```bash
curl -X POST https://api.ararahq.com/v1/messages \
  -H "Authorization: Bearer ara_live_xxx" \
  -d '{
    "receiver": "whatsapp:+5511999999999",
    "templateName": "fatura_disponivel",
    "templateVariables": [
      "https://meusite.com/fatura.pdf",
      "João",
      "Janeiro"
    ]
  }'
```

Suportado: imagem (JPEG/PNG, max 5MB), vídeo (MP4 H.264, max 16MB, max 16s), PDF (max 100MB, max 100 páginas).

## Mensagem agendada

```bash
curl -X POST https://api.ararahq.com/v1/messages \
  -H "Authorization: Bearer ara_live_xxx" \
  -d '{
    "receiver": "whatsapp:+5511999999999",
    "templateName": "lembrete_consulta",
    "templateVariables": ["amanhã às 14h"],
    "scheduled_at": "2026-05-19T14:00:00-03:00"
  }'
```

Mensagem fica em `status: SCHEDULED` até o horário.

## Mensagem interativa (janela aberta)

Botões de resposta rápida ou listas selecionáveis dentro da janela 24h:

```bash
curl -X POST https://api.ararahq.com/v1/messages \
  -d '{
    "receiver": "whatsapp:+5511999999999",
    "interactive": {
      "type": "button",
      "body": "Qual a sua dúvida?",
      "buttons": [
        { "id": "rastreio", "title": "Rastrear pedido" },
        { "id": "troca", "title": "Trocar produto" },
        { "id": "humano", "title": "Falar com humano" }
      ]
    }
  }'
```

Quando cliente clica, vem como webhook `message.received` com `type: button` e o `id` clicado.

## Lote — até 1000 mensagens

`POST /v1/messages/batch`

```bash
curl -X POST https://api.ararahq.com/v1/messages/batch \
  -H "Authorization: Bearer ara_live_xxx" \
  -H "Idempotency-Key: batch-pedidos-20260518" \
  -d '{
    "templateName": "pedido_confirmado",
    "messages": [
      { "receiver": "whatsapp:+5511999999999", "templateVariables": ["João", "ARA-1"] },
      { "receiver": "whatsapp:+5511888888888", "templateVariables": ["Maria", "ARA-2"] }
    ]
  }'
```

Resposta (`202 Accepted`):

```json
{
  "batchId": "ara_batch_xyz",
  "templateName": "pedido_confirmado",
  "total": 2,
  "accepted": 2,
  "totalCost": 0.24,
  "messages": [
    {
      "id": "ara_msg_a",
      "receiver": "whatsapp:+5511999999999",
      "status": "PENDING",
      "cost": 0.12
    },
    {
      "id": "ara_msg_b",
      "receiver": "whatsapp:+5511888888888",
      "status": "PENDING",
      "cost": 0.12
    }
  ]
}
```

Diferença pra `POST /v1/messages` individual: você manda o `templateName` **uma vez** no envelope, cada mensagem só precisa do `receiver` e `templateVariables`.

Pra > 1000 destinatários, use [Campanhas](/api-reference/endpoint/v2/campaigns).

## Consultar status

`GET /v1/messages/{id}`

```bash
curl https://api.ararahq.com/v1/messages/ara_msg_k9Bf2XqP \
  -H "Authorization: Bearer ara_live_xxx"
```

Resposta (`200 OK`):

```json
{
  "id": "ara_msg_k9Bf2XqP",
  "status": "DELIVERED",
  "mode": "LIVE",
  "sender": "whatsapp:+5511888888888",
  "receiver": "whatsapp:+5511999999999",
  "cost": 0.12
}
```

Em produção, **prefira webhook** (`message.status`) em vez de polling. Ver [Eventos](/webhooks/events).

## Parâmetros do POST

| Campo | Tipo | Obrigatório | Descrição |
|---|---|---|---|
| `receiver` | string | sim | E.164 com prefixo `whatsapp:` (`whatsapp:+5511999999999`) |
| `templateName` | string | * | Nome do template aprovado pela Meta |
| `templateVariables` | string[] | não | Array posicional. 1º elemento = mídia se template tem header |
| `body` | string | * | Texto livre — apenas dentro da janela 24h |
| `interactive` | object | * | Mensagem com botões/lista — apenas dentro da janela 24h |
| `location` | object | * | Localização — apenas dentro da janela 24h |
| `reaction` | object | * | Reação a mensagem prévia (emoji) |
| `replyTo` | string | não | ID de mensagem original pra citar (reply) |
| `scheduled_at` | string (ISO 8601) | não | Agendamento futuro |

`*` — exatamente um entre `templateName`, `body`, `interactive`, `location`, `reaction`.

## Status

Respostas síncronas (`POST`/`GET /v1/messages`) retornam o status em **UPPERCASE**:

| Status | Significado |
|---|---|
| `PENDING` | Criada, aguardando worker |
| `SCHEDULED` | Agendada, aguardando horário |
| `PROCESSING` | Worker pegou |
| `SENT` | Provider confirmou envio |
| `DELIVERED` | Chegou no destinatário (2 ticks cinza) |
| `READ` | Cliente leu (2 ticks azuis) |
| `FAILED` | Falhou definitivamente |
| `CANCELED` | Cancelada antes do envio |

<Note>
  Payloads de **webhook** (`message.status_updated`) usam os mesmos estados em **minúsculo** (`queued`, `sent`, `delivered`, `read`, `failed`, `scheduled`, `processing`, `canceled`).
</Note>

## Erros

Erros de negócio saem no envelope canônico (`{ "error": { "code", "message", "details" } }`). Validação de body (`@Valid`) e autenticação usam o envelope **simplificado** do Spring (sem `code`) — ramifique pelo status. Ver [Erros](/erros).

| HTTP | Código / Envelope | Quando |
|---|---|---|
| `400` | simplificado (sem `code`) | `receiver` ausente ou E.164 inválido (`@Valid`). |
| `400` | `INVALID_JSON` | JSON malformado. |
| `401` | simplificado (sem `code`) | Chave inválida, revogada ou expirada. |
| `402` | `INSUFFICIENT_FUNDS` | Saldo insuficiente. |
| `403` | `ACCOUNT_NOT_ACTIVATED` / `NO_DEDICATED_NUMBER` / `PLAN_FEATURE_LOCKED` | Conta em onboarding, sem número dedicado, ou feature travada no plano. |
| `409` | `CONFLICT` | Template não encontrado no envio, ou conflito de estado. |
| `422` | `UNPROCESSABLE_ENTITY` | Regra de negócio: template não aprovado, variáveis incompatíveis, ou fora da janela 24h (`details.diagnostic: "63016"`). |
| `422` | `NUMBER_BLOCKED` | Destinatário em opt-out. |
| `429` | `ORGANIZATION_SUSPENDED` | Org suspensa por taxa de falha alta; ou rate limit global (respeite `Retry-After`). |

## Idempotência

POST aceita header `Idempotency-Key`:

```bash
-H "Idempotency-Key: order-confirm-abc123"
```

Mesma chave + mesmo body em 24h retorna a resposta original. Recomendado em **qualquer caller de produção**.
