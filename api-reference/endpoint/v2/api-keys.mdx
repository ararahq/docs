Gerencie as chaves que suas aplicações usam pra autenticar na API.

**Base URL:** `https://api.ararahq.com/api`

<Warning>
  A chave plain text é exibida **uma única vez**, no momento da criação (campo `plainTextKey`). A Arara não armazena o valor completo — se perder, crie uma nova e revogue a antiga.
</Warning>

## Criar uma API key

`POST /v1/api-keys`

```bash
curl -X POST "https://api.ararahq.com/api/v1/api-keys?mode=LIVE" \
  -H "Authorization: Bearer ara_live_xxx" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "backend-produção",
    "scope": "ADMIN"
  }'
```

<Note>
  `mode` vai como **query param**, não no body. Aceita `LIVE` (default) ou `TEST`.
</Note>

### Body

| Campo | Tipo | Obrigatório | Descrição |
|---|---|---|---|
| `name` | string | sim | Nome humano (ex: "Site WordPress"). Vazio é rejeitado. |
| `scope` | string | não | `ADMIN` (default), ou outros escopos da sua org. |
| `expiresAt` | string (ISO 8601) | não | Data de expiração. Omitir = nunca expira. |
| `ipAllowlist` | string | não | IPs autorizados (string CSV). **Requer plano Business+**. |

### Query params

| Param | Tipo | Default | Descrição |
|---|---|---|---|
| `mode` | string | `LIVE` | `LIVE` ou `TEST` |

Resposta (`201 Created`):

```json
{
  "plainTextKey": "ara_live_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"
}
```

A response é **minimal por design** — só o valor plain text da chave nova. Pra metadados (id, prefix, criado em, etc), faça `GET /v1/api-keys` depois.

<Warning>
  Conta ainda em onboarding (`accountStatus: TESTING`) não pode criar chave **LIVE** — retorna `ACCOUNT_NOT_ACTIVATED`. Pode criar TEST.
</Warning>

## Listar chaves ativas

`GET /v1/api-keys`

Retorna **array direto** com todas as chaves ativas (não-revogadas) do user autenticado.

```bash
curl https://api.ararahq.com/api/v1/api-keys \
  -H "Authorization: Bearer ara_live_xxx"
```

Resposta (`200 OK`):

```json
[
  {
    "id": "7f8a1c2d-4e5f-6789-abcd-ef0123456789",
    "name": "backend-produção",
    "prefix": "ara_live",
    "lastFour": "a3f9",
    "mode": "LIVE",
    "scope": "ADMIN",
    "createdAt": "2026-03-01T10:00:00Z",
    "lastUsedAt": "2026-05-18T14:25:00Z",
    "expiresAt": null,
    "expired": false,
    "ipAllowlist": null,
    "rotatedFromId": null
  }
]
```

### Campos

| Campo | Tipo | Descrição |
|---|---|---|
| `id` | UUID | Identificador interno da chave |
| `name` | string | Nome humano (`"Sem nome"` se não definido) |
| `prefix` | string | Prefixo (`ara_live` ou `ara_test`) |
| `lastFour` | string | Últimos 4 caracteres da chave (não-secreto) |
| `mode` | string | `LIVE` ou `TEST` |
| `scope` | string | Escopo de permissões |
| `createdAt` | ISO 8601 | Quando foi criada |
| `lastUsedAt` | ISO 8601 \| null | Último uso. `null` se nunca usada |
| `expiresAt` | ISO 8601 \| null | Quando expira. `null` se sem expiração |
| `expired` | boolean | `true` se `expiresAt` já passou |
| `ipAllowlist` | string \| null | IPs autorizados, se configurado |
| `rotatedFromId` | UUID \| null | ID da chave original se foi rotacionada |

## Atualizar chave

`PATCH /v1/api-keys/{id}`

```bash
curl -X PATCH https://api.ararahq.com/api/v1/api-keys/7f8a1c2d-4e5f-6789-abcd-ef0123456789 \
  -H "Authorization: Bearer ara_live_xxx" \
  -d '{
    "name": "novo-nome",
    "ipAllowlist": "52.10.0.0/16"
  }'
```

Body aceita os mesmos campos do POST, mais:

| Campo | Tipo | Descrição |
|---|---|---|
| `clearExpiration` | boolean | `true` remove `expiresAt` |
| `clearIpAllowlist` | boolean | `true` remove `ipAllowlist` |

Resposta: **`204 No Content`** sem body.

## Rotacionar chave

`POST /v1/api-keys/{id}/rotate`

Cria chave nova com mesmas permissões e marca a anterior pra expirar em 24h.

```bash
curl -X POST https://api.ararahq.com/api/v1/api-keys/{id}/rotate \
  -H "Authorization: Bearer ara_live_xxx"
```

Resposta (`200 OK`) traz a nova chave (campo `plainTextKey`) e o ID da chave que foi rotacionada.

Útil pra rotação periódica (PCI, SOC2) ou quando suspeita de vazamento.

## Revogar chave

`DELETE /v1/api-keys/{id}`

```bash
curl -X DELETE https://api.ararahq.com/api/v1/api-keys/7f8a1c2d-4e5f-6789-abcd-ef0123456789 \
  -H "Authorization: Bearer ara_live_xxx"
```

Resposta: **`204 No Content`** sem body.

Próxima requisição com a chave revogada retorna `401`.

## Estatísticas de uso

`GET /v1/api-keys/{id}/stats`

```bash
curl https://api.ararahq.com/api/v1/api-keys/{id}/stats \
  -H "Authorization: Bearer ara_live_xxx"
```

Resposta:

```json
{
  "totalLast7d": 1240,
  "totalLast30d": 5840,
  "successLast30d": 5712,
  "successRateLast30d": 0.978
}
```

| Campo | Descrição |
|---|---|
| `totalLast7d` | Total de requests nos últimos 7 dias |
| `totalLast30d` | Total de requests nos últimos 30 dias |
| `successLast30d` | Quantos retornaram 2xx |
| `successRateLast30d` | Taxa de sucesso (0..1) ou `null` se zero requests |

## Histórico de eventos

`GET /v1/api-keys/{id}/events`

Retorna eventos de auditoria da chave (criação, rotação, revogação, mudança de IP allowlist).

```bash
curl https://api.ararahq.com/api/v1/api-keys/{id}/events \
  -H "Authorization: Bearer ara_live_xxx"
```

Resposta:

```json
[
  {
    "id": "abc...",
    "type": "CREATED",
    "actorUserId": "user-uuid",
    "metadata": "{\"name\":\"backend-produção\"}",
    "createdAt": "2026-03-01T10:00:00Z"
  }
]
```

`metadata` vem como **string JSON** (você dá `JSON.parse` no cliente).

## IP allowlist (Business+)

Plano Business+ permite restringir IPs autorizados pra cada chave. Aceita IP único ou CIDR:

```bash
curl -X PATCH https://api.ararahq.com/api/v1/api-keys/{id} \
  -H "Authorization: Bearer ara_live_xxx" \
  -d '{ "ipAllowlist": "52.10.0.0/16" }'
```

Chamada de IP fora da lista retorna `401`.

<Note>
  Hoje `ipAllowlist` é uma **string única** (pode ser CSV se vários IPs). Não é array.
</Note>

## Erros

| HTTP | Quando |
|---|---|
| `400` | `name` vazio ou `mode` inválido |
| `401` | Chave inválida ou ausente |
| `403` | Conta em TESTING tentando criar LIVE (`ACCOUNT_NOT_ACTIVATED`); ou IP allowlist sem plano Business+ |
| `404` | Chave não existe ou não pertence ao user |

## Segurança

- Nunca exponha a chave em código frontend, repositório público ou log
- Use variáveis de ambiente (`ARARA_API_KEY`)
- Crie chaves separadas por ambiente (dev, staging, prod)
- Rotacione periodicamente (90 dias é razoável)
- IP allowlist quando possível (Business+)
- Revogue imediato quando dev sai do time
