A Arara aceita **dois modelos de autenticação** dependendo de quem está chamando.

| Modelo | Quem usa | Header |
|---|---|---|
| **API key Bearer** | Backend, scripts, SDKs, CLI, MCP | `Authorization: Bearer ara_live_...` |
| **Firebase JWT** | Dashboard, sessão de usuário, login | `Authorization: Bearer <firebase_id_token>` |

A maioria dos endpoints da Arara usa **API key**. Firebase JWT é usado apenas em endpoints de sessão (`/auth/*`, `/users/*`, `/v1/profile/*`).

## API key

Formato:

```
ara_<modo>_<32 caracteres>
```

| Prefixo | Modo | Comportamento |
|---|---|---|
| `ara_live_` | **LIVE** (produção) | Mensagens entregues. Créditos consumidos. |
| `ara_test_` | **TEST** (sandbox) | Mensagens simuladas. Gratuito, ilimitado. |

Use **TEST** pra desenvolvimento e CI. **LIVE** só em produção.

### Como obter

**Via Dashboard**: primeiro login gera automaticamente 1 chave LIVE e 1 chave TEST. Aparecem **uma vez**. Copie e guarde.

**Via API** (cliente já autenticado pode criar mais):

```bash
curl -X POST https://api.ararahq.com/v1/api-keys \
  -H "Authorization: Bearer ara_live_xxx" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "backend-produção",
    "mode": "LIVE",
    "ipAllowlist": ["52.10.0.0/16"]
  }'
```

Resposta `201 Created`:

```json
{
  "id": "ara_key_abc1234",
  "name": "backend-produção",
  "key": "ara_live_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx",
  "mode": "LIVE",
  "lastFour": "xxxx",
  "createdAt": "2026-05-18T14:30:00Z"
}
```

<Warning>
  O campo `key` aparece **apenas na criação**. Listagens posteriores retornam só `lastFour` (mascarado). Se perder, crie outra.
</Warning>

### Listar chaves ativas

```bash
curl "https://api.ararahq.com/v1/api-keys?mode=LIVE" \
  -H "Authorization: Bearer ara_live_xxx"
```

Retorna chaves ativas, com `lastFour` pra identificar sem expor o valor.

### Rotacionar

Cria chave nova com mesmas permissões e marca a anterior pra expirar em 24h:

```bash
curl -X POST https://api.ararahq.com/v1/api-keys/{id}/rotate \
  -H "Authorization: Bearer ara_live_xxx"
```

Útil pra rotação periódica (PCI, SOC2) ou quando suspeita de vazamento.

### Revogar

Invalida imediatamente:

```bash
curl -X DELETE https://api.ararahq.com/v1/api-keys/{id} \
  -H "Authorization: Bearer ara_live_xxx"
```

Próxima requisição com essa chave retorna `401`.

## IP allowlist (BUSINESS+)

Restringe quais IPs podem usar a chave:

```bash
curl -X PATCH https://api.ararahq.com/v1/api-keys/{id} \
  -H "Authorization: Bearer ara_live_xxx" \
  -d '{ "ipAllowlist": "52.10.0.0/16,203.0.113.5" }'
```

`ipAllowlist` é string CSV (IPs únicos ou CIDR, separados por vírgula). Chamada de IP fora da lista retorna `401`. Logs ficam em `/v1/api-keys/{id}/events`.

## Firebase JWT

Endpoints `/auth/*`, `/users/*` e `/v1/profile/*` exigem token Firebase. Fluxo:

```
1. Cliente (web/mobile) faz login no Firebase Auth
2. Pega ID token: firebase.auth().currentUser.getIdToken()
3. Envia no header: Authorization: Bearer <id_token>
4. Arara valida com Google, devolve sessão Arara + apiKey
```

Detalhe em [Sessão e Usuário](/api-reference/endpoint/v2/auth).

<Note>
  Você **não precisa do Firebase** pra usar a API — só pra dashboard. Integrações backend usam API key direto.
</Note>

## Segurança — checklist

- [ ] Chave **nunca** no código (use env var, secrets manager)
- [ ] Chave **nunca** no frontend (sempre via seu backend)
- [ ] Chave **nunca** em log (logger redacted: `ara_live_****`)
- [ ] Chave **nunca** em URL (header ou body — nunca query string)
- [ ] Chaves separadas por ambiente (dev, staging, prod)
- [ ] Rotação periódica (90 dias é razoável)
- [ ] IP allowlist quando possível
- [ ] Revoga imediato quando dev sai do time

## Erros de auth

```json
{
  "error": {
    "code": "INVALID_API_KEY",
    "message": "The provided API key is invalid or has been revoked."
  }
}
```

| Código | HTTP | Causa |
|---|---|---|
| `MISSING_API_KEY` | 401 | Header `Authorization` ausente |
| `INVALID_API_KEY` | 401 | Chave malformada, revogada ou inexistente |
| `INVALID_API_KEY_MODE` | 401 | Chave TEST tentando agir em LIVE (ou vice-versa) |
| `IP_NOT_ALLOWED` | 401 | Allowlist bloqueou IP de origem |
| `FORBIDDEN` | 403 | Chave válida mas sem permissão pro endpoint |
